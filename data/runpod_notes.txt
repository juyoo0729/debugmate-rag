# DebugMate RAG RunPod 사용 노트

이 문서는 AI 학습 과정에서 RunPod를 사용할 때 자주 헷갈리는 개념, 저장공간 문제, GPU 선택 문제, 비용 관리 방법을 정리한 RAG 검색용 문서이다.

# 주제 1: RunPod는 로컬 GPU를 가져오는 방식이 아니다

## 상황

사용자가 RunPod를 사용하면 자신의 로컬 PC GPU 자원을 가져와 쓰는 것으로 오해할 수 있다.

## 설명

RunPod는 로컬 PC의 GPU를 사용하는 방식이 아니다.

RunPod는 원격 GPU 서버를 빌려서 사용하는 클라우드 GPU 서비스이다.

즉, 로컬 PC는 접속용 장치이고, 실제 계산은 RunPod 서버에서 수행된다.

## 구조

```text
로컬 PC
→ VS Code, SSH, JupyterLab 접속
→ RunPod 원격 서버
→ GPU 계산 수행
```

## 주의사항

로컬 PC 성능이 낮아도 RunPod GPU 서버를 사용하면 대형 모델 실행이나 학습이 가능하다.

하지만 로컬 파일이 자동으로 RunPod에 저장되는 것은 아니므로 GitHub, 업로드, 다운로드, 볼륨 관리가 필요하다.

## 관련 태그

runpod, gpu, remote_server, cloud_gpu

# 주제 2: RunPod 저장공간 종류

## 상황

RunPod에서 작업한 파일이 Pod를 종료하거나 삭제한 뒤 사라질 수 있다.

## 원인

RunPod 저장공간은 종류에 따라 보존 방식이 다르다.

## 저장공간 종류

### 1. Container Disk

Container Disk는 컨테이너 내부 시스템 영역이다.

예:

```text
/usr/local
/root
```

특징:

* Pod 삭제 시 사라질 수 있다.
* 패키지 설치나 임시 파일이 저장될 수 있다.
* 장기 보관용으로 적합하지 않다.

### 2. Volume Disk

Volume Disk는 보통 `/workspace`에 연결되는 작업 공간이다.

예:

```text
/workspace
```

특징:

* Pod 생명주기 동안 작업 파일을 유지할 수 있다.
* Stop/Start에서는 유지될 수 있다.
* Terminate 또는 설정에 따라 삭제될 수 있으므로 중요한 코드는 GitHub에 백업해야 한다.

### 3. Network Volume

Network Volume은 Pod와 독립적으로 유지되는 영구 저장공간이다.

특징:

* Pod를 삭제해도 데이터가 유지된다.
* 여러 Pod에서 재사용할 수 있다.
* 비용이 발생한다.
* 특정 데이터센터에 묶일 수 있어 선택 가능한 GPU가 제한될 수 있다.

## 결론

초기 실습 단계에서는 Network Volume을 바로 만들기보다 GitHub로 코드 관리하고, 필요한 파일만 로컬에 백업하는 전략이 안전하다.

## 관련 태그

runpod, storage, workspace, network_volume, volume_disk

# 주제 3: RunPod에서는 /workspace 아래에 프로젝트를 만드는 것이 좋다

## 상황

RunPod에서 JupyterLab, SSH, VS Code Remote-SSH로 접속해 작업할 때 파일 저장 위치가 헷갈릴 수 있다.

## 권장 방식

RunPod에서는 프로젝트를 `/workspace` 아래에 만든다.

예:

```bash
cd /workspace
git clone https://github.com/juyoo0729/debugmate-rag.git
cd debugmate-rag
```

또는 새 프로젝트라면:

```bash
cd /workspace
mkdir debugmate-rag
cd debugmate-rag
```

## 이유

`/workspace`는 RunPod 작업 공간으로 사용되는 경우가 많고, 프로젝트 파일을 관리하기 쉽다.

반대로 `/root`, `/tmp`, `/usr/local` 등에 프로젝트를 만들면 Pod 설정에 따라 파일이 사라지거나 찾기 어려워질 수 있다.

## 주의사항

중요한 코드는 반드시 GitHub에 push해야 한다.

RunPod 저장공간은 설정과 Pod 상태에 따라 유지 여부가 달라질 수 있다.

## 관련 태그

runpod, workspace, github, project_path

# 주제 4: Network Volume은 언제 필요한가

## 상황

사용자가 RunPod에서 Network Volume을 만들지 고민한다.

## 설명

Network Volume은 Pod와 독립적으로 데이터를 보관하는 저장공간이다.

대용량 모델, 데이터셋, 체크포인트를 장기간 보관하려면 유용하다.

## 필요한 경우

Network Volume이 필요한 경우는 다음과 같다.

1. 대용량 모델 파일을 반복해서 다운로드하고 싶지 않을 때
2. 학습 체크포인트를 장기간 보관해야 할 때
3. 여러 Pod에서 같은 데이터를 재사용해야 할 때
4. Pod를 삭제해도 데이터가 유지되어야 할 때

## 아직 필요하지 않은 경우

초기 실습이나 작은 RAG 프로젝트에서는 꼭 필요하지 않다.

특히 OpenAI API 기반 RAG는 embedding과 LLM 계산이 외부 API에서 수행되므로 RunPod GPU와 대용량 저장공간이 필수는 아니다.

## 주의사항

Network Volume은 비용이 발생한다.

또한 특정 데이터센터에 묶여 사용할 수 있는 GPU 종류가 제한될 수 있다.

따라서 최종 프로젝트의 모델과 데이터 규모가 확정된 뒤 만드는 것이 좋다.

## 관련 태그

network_volume, storage_cost, runpod, gpu_limit

# 주제 5: RunPod 비용 관리 전략

## 상황

RunPod 비용이 예상보다 많이 나올 수 있다.

## 원인

GPU Pod를 켜둔 상태로 오래 두거나, Network Volume을 큰 용량으로 만들면 비용이 계속 발생할 수 있다.

## 비용 관리 방법

1. 작업이 끝나면 Pod를 중지하거나 삭제한다.
2. 장기 보관이 필요한 파일은 GitHub나 로컬에 백업한다.
3. Network Volume은 필요한 시점에만 만든다.
4. 큰 모델을 쓰기 전에는 예상 VRAM과 저장공간을 계산한다.
5. 실습 초기에는 작은 GPU 또는 로컬 환경으로 먼저 검증한다.

## 추천 전략

초기 단계:

```text
로컬 PC + GitHub + OpenAI API
```

중간 단계:

```text
RunPod JupyterLab + GitHub
```

고급 단계:

```text
RunPod GPU + Network Volume + 오픈소스 LLM
```

## 관련 태그

runpod, cost, gpu_cost, network_volume, budget

# 주제 6: OpenAI API 기반 RAG는 RunPod GPU를 많이 사용하지 않는다

## 상황

사용자가 RAGAS 평가나 OpenAIEmbeddings를 RunPod GPU에서 돌리면 GPU가 많이 사용된다고 생각할 수 있다.

## 설명

OpenAIEmbeddings와 ChatOpenAI는 OpenAI 서버에서 계산된다.

따라서 RunPod GPU 자원을 많이 사용하지 않는다.

RunPod에서는 주로 다음 작업을 수행한다.

* Python 코드 실행
* 문서 로드
* chunk 분할
* API 호출
* 결과 저장

실제 embedding 계산과 LLM 답변 생성은 OpenAI API 서버에서 수행된다.

## 결론

OpenAI API 기반 RAG 실습은 꼭 RunPod GPU가 없어도 로컬 PC에서 가능하다.

RunPod GPU가 필요한 경우는 오픈소스 LLM을 직접 실행하거나, 로컬 embedding 모델을 GPU로 돌릴 때이다.

## 관련 태그

openai_api, runpod, gpu_usage, embeddings, rag

# 주제 7: RunPod에서 오픈소스 LLM을 사용할 때 GPU가 중요하다

## 상황

사용자가 Qwen, Llama, Mistral 같은 오픈소스 LLM을 RunPod에서 실행하려고 한다.

## 설명

오픈소스 LLM을 직접 실행할 때는 GPU VRAM이 중요하다.

모델 크기에 따라 필요한 VRAM이 달라진다.

## 예시

```text
7B 모델
→ RTX 4090, A40, L40S급에서 비교적 실험 가능

14B 모델
→ 더 많은 VRAM이 필요하며, 양자화 여부에 따라 가능 여부가 달라짐

70B 모델
→ A100, H100급 또는 멀티 GPU 구성이 필요할 수 있음
```

## 주의사항

모델 파일 크기, KV cache, batch size, context length에 따라 실제 VRAM 사용량은 달라진다.

초기에는 작은 모델로 검증한 뒤 큰 모델로 넘어가는 것이 좋다.

## 관련 태그

runpod, open_source_llm, vllm, qwen, llama, gpu_vram

# 주제 8: RunPod 작업 흐름 추천

## 권장 작업 흐름

### 1단계: 로컬에서 코드 작성

로컬 PC에서 VS Code로 코드를 작성한다.

```text
C:\AI_study\debugmate-rag
```

GitHub에 push한다.

```powershell
git add .
git commit -m "Update project"
git push
```

### 2단계: RunPod에서 clone

RunPod에서 프로젝트를 clone한다.

```bash
cd /workspace
git clone https://github.com/juyoo0729/debugmate-rag.git
cd debugmate-rag
```

### 3단계: RunPod에서 환경 설치

```bash
python -m venv .venv
source .venv/bin/activate
python -m pip install -r requirements.txt
```

### 4단계: 실험 실행

JupyterLab이나 Streamlit으로 실험한다.

```bash
streamlit run app.py
```

### 5단계: 결과 저장

결과 파일을 GitHub에 push하거나 로컬로 다운로드한다.

## 관련 태그

runpod, github, workflow, jupyterlab, streamlit

# 주제 9: RunPod에서 저장이 안 된다고 느껴질 때 확인할 것

## 상황

RunPod에서 작업한 파일이 사라졌다고 느껴진다.

## 확인할 것

1. 파일을 `/workspace` 아래에 저장했는가?
2. Pod를 Stop한 것인가, Terminate한 것인가?
3. Network Volume을 사용했는가?
4. GitHub에 push했는가?
5. JupyterLab에서 현재 작업 디렉터리가 어디인가?

## 해결 방법

중요한 코드는 반드시 GitHub에 push한다.

```bash
git status
git add .
git commit -m "Save RunPod work"
git push
```

데이터 파일이나 결과 파일은 로컬로 다운로드하거나 Network Volume에 저장한다.

## 주의사항

RunPod는 클라우드 서버이므로 로컬 PC처럼 모든 파일이 영구 보존된다고 생각하면 안 된다.

## 관련 태그

runpod, file_loss, workspace, github, backup

# 주제 10: DLthon에서 RunPod 사용 전략

## 상황

DLthon MVP를 만들 때 RunPod를 꼭 써야 하는지 고민한다.

## 결론

DebugMate RAG MVP는 OpenAI API 기반으로 만들 수 있으므로 RunPod GPU가 필수는 아니다.

로컬 PC에서 Streamlit과 Chroma로 충분히 데모 가능하다.

## RunPod가 필요한 경우

다음과 같은 확장을 할 때 RunPod가 필요할 수 있다.

1. 오픈소스 LLM 직접 서빙
2. vLLM 실행
3. 로컬 embedding 모델 GPU 실행
4. 대용량 문서 처리
5. 모델 fine-tuning

## MVP 전략

DLthon에서는 다음 전략이 적절하다.

```text
로컬 PC에서 MVP 완성
→ GitHub 정리
→ 필요하면 RunPod에서 재현 가능하게 준비
```

## 관련 태그

dlthon, runpod, mvp, local_demo, github
